## Context

当前 Tab/Shift+Tab 焦点导航在 `focus_index = None` 时固定选择第一个/最后一个元素。用户已滚动到文档中间时，按 Tab 会跳到文档顶部，需要多次按键才能回到当前阅读位置。

核心代码位于 `src/app.rs` 的 `handle_key` 方法中 `KeyCode::Tab` 和 `KeyCode::BackTab` 分支。

## Goals / Non-Goals

**Goals:**
- Tab 首次聚焦时，选择当前可视区域内或之后的第一个可聚焦元素
- Shift+Tab 首次聚焦时，选择当前可视区域内或之前的最后一个可聚焦元素
- 边界情况：可视区域前后都没有元素时，选择最近的元素

**Non-Goals:**
- 不改变已有焦点时的 Tab/Shift+Tab 循环逻辑
- 不添加新的快捷键
- 不改变焦点样式或渲染逻辑

## Decisions

### Decision 1: 基于 scroll 位置查找起始焦点

**选择**: 使用二分查找或线性扫描，找到第一个 `position >= scroll` 的可聚焦元素。

**理由**: 实现简单，时间复杂度 O(log n) 或 O(n)，对于可聚焦元素数量来说完全可接受。

**替代方案**:
- 记录上次焦点位置 → 增加状态复杂度，且不符合用户预期
- 基于鼠标位置 → 终端环境不适用

### Decision 2: 边界处理策略

**选择**:
- Tab：可视区域后无元素 → 选最后一个（即离当前屏幕最近的下方元素不存在，说明都在上方，选最后一个）
- Shift+Tab：可视区域前无元素 → 选第一个

**理由**: 用户已滚动到极端位置时，聚焦到最接近当前屏幕的元素是最自然的行为。

## Risks / Trade-offs

**[Risk] 可视区域判定不准确** → 使用现有的 `scroll` 和 `height` 计算，与渲染逻辑一致，无额外风险。

**[Risk] 测试用例需要更新** → 现有测试 `tab_focuses_first_image` 等需要修改，添加新测试覆盖可视区域感知逻辑。
