## Context

当前 `src/xhtml.rs` 中的 HTML 解析器使用 `quick-xml` 库逐标签解析 HTML，并将其转换为终端可显示的 `LineContent`。现有的处理逻辑：

- **已支持**：标题（h1-h6）、段落（p）、列表（ul/ol/li）、链接（a）、图片（img）、粗体/斜体（strong/em）、引用（blockquote）
- **部分支持**：代码（code、pre）- 有基本的样式处理，但可能丢失缩进和格式
- **未支持**：表格（table、thead、tbody、tr、th、td）- 完全没有处理，标签内容会被当作普通文本

问题根源：
1. 表格标签没有被 `match tag` 匹配，导致标签被忽略但内容被保留，格式混乱
2. 代码块在 `in_pre` 模式下可能没有正确处理换行和缩进
3. 终端显示需要特殊的表格渲染方式（使用 ASCII/Unicode 边框字符）

## Goals / Non-Goals

**Goals:**
- 实现基本的表格渲染，将 HTML 表格转换为终端友好的文本表格
- 改进代码块的解析，保留缩进、换行和空格
- 保持代码简洁，不引入复杂的表格布局算法
- 提供合理的终端表格样式（使用 Unicode 边框字符）

**Non-Goals:**
- 不支持复杂的表格合并（colspan、rowspan）- 过于复杂
- 不支持响应式表格布局 - 终端宽度固定
- 不实现完整的 HTML 表格规范 - 只处理常见场景
- 不添加语法高亮 - 需要额外的依赖和配置

## Decisions

### 1. 表格渲染方案：简单文本表格

**决策**：使用 Unicode 边框字符（┌─┬┐│├┼┤└─┴┘）渲染表格，每列宽度根据内容自动计算。

**理由**：
- Unicode 边框字符在大多数终端中显示良好
- 自动计算列宽可以适应不同内容
- 实现相对简单，不需要复杂的布局算法

**替代方案**：
- **Markdown 风格表格**（使用 | 和 -）：不够美观
- **固定宽度表格**：无法适应不同内容
- **支持 colspan/rowspan**：实现复杂度太高

### 2. 表格解析策略：两遍处理

**决策**：先收集表格数据到内存，计算列宽后再渲染。

**理由**：
- 需要知道所有单元格的内容才能计算合适的列宽
- 两遍处理（解析 → 存储 → 渲染）逻辑清晰
- 内存占用可控（表格通常不大）

### 3. 代码块处理：保留原始格式

**决策**：在 `in_pre` 模式下，不 trim 文本，保留所有空格和换行。

**理由**：
- 代码的缩进是语义的一部分
- `quick-xml` 的 `trim_text(false)` 配置已经设置了不 trim
- 需要确保 pre 标签内的文本被完整保留

### 4. 表格宽度限制

**决策**：如果表格总宽度超过终端宽度（假设 80 列），按比例压缩列宽或截断内容。

**理由**：
- 终端宽度有限，需要适配
- 简单的截断比复杂的换行更容易实现
- 可以添加提示说明表格被截断

## Risks / Trade-offs

**[Risk] 表格渲染可能不够美观**
简单的文本表格在终端中可能不如 HTML 表格美观。
→ **Mitigation**: 使用 Unicode 边框字符提升视觉效果；接受"足够好"而不是"完美"。

**[Risk] 宽表格可能超出终端宽度**
如果表格列数多或内容长，可能超出终端宽度。
→ **Mitigation**: 实现简单的列宽压缩或截断逻辑；添加提示信息。

**[Risk] 不支持 colspan/rowspan**
复杂表格可能无法正确渲染。
→ **Mitigation**: 明确文档说明不支持；将 colspan/rowspan 的单元格当作普通单元格处理。

**[Risk] 代码块可能仍然有格式问题**
某些代码块可能使用了特殊的 HTML 结构（如 `<div class="code">`）。
→ **Mitigation**: 只处理标准的 pre/code 标签；对于非标准结构，建议用户使用其他工具。

**[Trade-off] 实现简单性 vs 功能完整性**
选择不支持 colspan/rowspan 和复杂布局，简化实现但降低兼容性。
→ **接受**：大多数网页表格是简单的，复杂的表格很少见。
